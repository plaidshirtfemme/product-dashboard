# Дизайн-система по канону — свод правил

> Исследование DASH-120 + DASH-121 (12.07.2026). Как строить и вести дизайн-систему
> по лучшим практикам команд 2026 года: оформление в Figma, токены, документация,
> синхронизация дизайн↔код. Источник правды правил — этот файл + CLAUDE.md.

## 0. Главный принцип: токены — единый источник правды

Канон 2026: **граф токенов** — единственный источник правды, чтобы дизайн и код
**не могли разойтись**. Не «Figma главная» и не «код главный» — токены посередине,
их читают обе стороны.

- Стандарт: **W3C DTCG v2025.10** (стабилен с окт. 2025). Один JSON-формат
  `$value`/`$type`, который Figma, Tokens Studio, Style Dictionary, Penpot,
  Supernova читают без конвертеров.
- Наш `dash_app/tokens/design_tokens.json` уже в этом формате (`$schema`,
  `$value`/`$type`) — правильный современный подход. **Гэп:** структура плоская,
  нужно привести к 3 уровням (ниже).

## 1. Три уровня токенов (primitive → semantic → component)

Канонная трёхслойка (74% команд строят DS на Figma Variables именно так):

| Уровень | Что | Пример |
|---|---|---|
| **Primitive** (raw) | сырые значения, без смысла | `blue-500 = #3B82F6`, `space-4 = 16px` |
| **Semantic** (aliases) | роль, ссылается на primitive | `text-primary → gray-900`, `surface-success → green-200` |
| **Component** | привязка к компоненту | `button-radius → radius-md`, `card-padding → space-4` |

Правило: код и Figma потребляют **semantic/component**, не primitive напрямую.
Смена темы = переназначение semantic-алиасов, primitive не трогаем.

## 2. Структура файлов в Figma

- **Отдельная команда/проект** под дизайн-систему, со своими правами.
- Разделение по стадиям: `Work in progress` / `In development` / `In production`.
- Три файла DS:
  1. **Foundations** — токены/variables, цвет, типографика, spacing, radius, тени.
  2. **Components** — библиотека компонентов.
  3. **Documentation** — как пользоваться (do/don't, анатомия, состояния).
- **Страницы = категории** (Buttons, Forms, Navigation…), компоненты — на артбордах.
- Каскад библиотек: сверху — универсальные стили/компоненты, вниз — специфичные.

⚠️ **Наш блокер (free-план):** публиковать библиотеки нельзя (нужен paid). На free
DS показываем **визуально** (компоненты разложены на странице + токены-таблицей),
но не как консумируемую published library. Для рекрутера визуальной подачи достаточно;
«настоящая» published DS — довод к апгрейду (см. DASH-118).

## 3. Variables (Figma) — три коллекции

Зеркалят три уровня токенов: **Primitive Collection** (raw hex/px) → **Semantic
Collection** (`text-primary`, `surface-success`) → **Component Collection**
(`button-radius`, `card-padding`). Режимы (modes) — для тем (light/dark).

## 4. Нейминг и документация

- Логичная структура, консистентный нейминг, регулярная поддержка.
- Нейминг токенов: `категория-роль-вариант` (`color-text-primary`, `space-inset-md`).
- Документация каждого компонента: анатомия, состояния, do/don't, когда использовать.
- Figma AI 2026 умеет массово переименовывать слои перед handoff и искать дубли
  визуальным поиском — использовать для чистоты файла.

## 5. Синхронизация дизайн ↔ код (пайплайн)

Канонный поток (DASH-121):

```
Figma Variables  ↔  Tokens Studio  ↔  design_tokens.json (DTCG)  →  код
                                                     │
                                        Style Dictionary 4 → CSS/Tailwind/TS
                                                     │
                                        Storybook preview → версионированный npm
```

- **Валидация:** JSON-schema против DTCG в CI (ловит дрейф формата).
- **Против дрейфа компонентов:** Code Connect — маппинг Figma-компонент → код-компонент.
- **Версионирование:** DS как версионированный пакет; изменения через контрибуции + review.
- **Ритуалы:** регулярный DS-review, contribution model (кто и как добавляет в DS).

### Наш Reflex-нюанс

У нас **нет Storybook/npm** — `tokens.py` читает `design_tokens.json` напрямую и
отдаёт в Reflex-компоненты. Значит:
- Style Dictionary **опционален** (пригодится, только если захотим экспорт в чистый CSS).
- Единый источник у нас = `design_tokens.json`; Tokens Studio синкает его с Figma
  Variables — это и есть наш мост дизайн↔код без Figma→code генерации.
- Code Connect у нас неприменим напрямую (Reflex, не React-компоненты в JS).

## 6. Подводные камни (за чем следить)

- **Плоские токены** (наш случай) — трудно менять темы и переиспользовать; привести к 3 уровням.
- **DS без потребителя** — если компоненты в Figma не связаны с кодом, расходятся;
  наш антидот — токены как единый источник + Tokens Studio.
- **Публикация library требует paid** — на free только визуальная подача.
- **Перфекционизм DS съедает время** (риск, названный Guzel) — на дедлайне делаем
  DS ровно настолько, чтобы она читалась каноном для рекрутера, не «идеальной».

## 7. Чек-лист «наша DS по канону»

- [ ] `design_tokens.json` → 3 уровня (primitive/semantic/component)
- [ ] Semantic-алиасы для тем (light/dark через modes)
- [ ] Нейминг `категория-роль-вариант`
- [ ] Figma: Foundations + Components + Docs (визуально, без publish на free)
- [ ] Tokens Studio синк `design_tokens.json` ↔ Figma Variables
- [ ] Страница-документация: анатомия/состояния/do-don't ключевых компонентов
- [ ] DS-артефакт для рекрутера: токены-таблица + компоненты + «single source of truth» нарратив
