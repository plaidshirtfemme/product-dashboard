# Спайк DASH-87 — сохранение логов сессий Claude Code

> Исследовано 12.07.2026 по реальным логам проекта. Как устроены логи сессий,
> как их читать и что с ними делать (в т.ч. для DASH-113 — бэкфилл changelog).

## 1. Где лежат и как называются

- Путь: `~/.claude/projects/<encoded-cwd>/<sessionId>.jsonl`
  Для нас: `C:\Users\guzel\.claude\projects\C--Users-guzel\*.jsonl`
- **Один файл = одна сессия**, имя файла = `sessionId` (UUID).
- Кодирование пути проекта: слэши/двоеточия → дефисы (`C:\Users\guzel` → `C--Users-guzel`).
- Сейчас 6 сессий.

## 2. Формат

- **JSONL** — одно событие на строку (валидный JSON).
- Типы событий: `user`, `assistant`, `system`, `attachment` + служебные
  (`mode`, `queue-operation`, `custom-title`, `ai-title`, `last-prompt`).
- Полезные поля верхнего уровня:
  - `timestamp` — время события (для сопоставления с git-коммитами и статусами);
  - `uuid` / `parentUuid` — дерево реплик (можно восстановить ход);
  - `sessionId`, `cwd`, `gitBranch` — контекст;
  - `message` / `content` — содержимое реплики;
  - `toolUseResult`, `toolUseID` — результаты вызовов инструментов;
  - `attributionMcpTool` / `attributionMcpServer` / `attributionSkill` — какой MCP/скилл вызван;
  - `permissionMode`, `version`.

## 4. Подходы к сохранению/использованию

1. **Снэпшот важных решений** → Obsidian/wiki. Полу-ручной: по ходу или скриптом
   вытащить decision-подобные фрагменты из `assistant`-событий.
2. **Извлечение в CLAUDE.md / changelog.** Пройти события, сопоставить по `timestamp`
   с git-коммитами → кто/когда/что решил.
3. **Связь с Jira-задачами.** `timestamp` событий ↔ `started`/`resolved` задач;
   `gitBranch` ↔ задача. Так можно реконструировать реальные даты смен статуса.
4. **Для DASH-113 (бэкфилл changelog):** полу-ручной скрипт читает эти файлы,
   находит по времени, когда задача менялась (упоминания ключа DASH-NN в репликах +
   git-коммиты), проставляет `started`/`resolved`. Не требует тяжёлой автоматизации.

## 5. Подводные камни

- **Приватность:** логи содержат пути, содержимое, иногда чувствительное → **не коммитить
  сырые логи**. Хранить только извлечённые выжимки.
- **Объём:** файл активной сессии — тысячи событий (наш ~2400+). Читать потоково.
- **Недокументированный формат:** официально Anthropic схему не фиксирует, поля могут
  меняться между версиями Claude Code — не строить хрупкую жёсткую зависимость.
- **Ротация не гарантирована:** нельзя рассчитывать, что логи вечны; для истории лучше
  снэпшотить важное в wiki/CLAUDE.md.

## 6. Рекомендация

Не строить тяжёлую автоматизацию сейчас. Для **DASH-113** достаточно полу-ручного
скрипта по этим 6 файлам + git-истории (сопоставление по времени). Ценное из логов
переносить выжимками в wiki/CLAUDE.md (мы это уже де-факто делаем через decision_note
в задачах). Сырые логи — не в репозиторий.
